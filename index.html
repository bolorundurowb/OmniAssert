Fluent assertions for modern .NET

Tests that read clearly and fail helpfully.

OmniAssert is a fluent assertion library for .NET 10 and C# 14. It helps you express test intent in plain language, collect related failures, compare complex objects, and use optional compile-time diagnostics for boolean expressions.

01. Overview

Project overview and value proposition

OmniAssert exists to make tests easier to read at the point of writing and easier to diagnose at the point of failure. Its fluent API keeps assertions close to the values being checked, while its failure messages capture the expression, actual value, expected value, and useful context.

Readable intent

Assertions start from the subject under test, for example user.Email.Verify().ToContain("@"). The result reads like plain English and stays consistent across strings, numbers, collections, objects, exceptions, files, dates, GUIDs, URIs, spans, and nullable values.

Better failures

Failures include captured caller expressions and formatted values. Complex collections are expanded, object equivalence failures show the path where data diverges, and ANSI colour support improves terminal output.

Modern diagnostics

The bundled Roslyn generator can enhance VerifyExpression() calls at compile time, so failed boolean expressions report operand values rather than a bare “condition was false”.

Provenance note: OmniAssert was created with substantial AI-assisted coding support. If you use it in production, security-sensitive, or compliance-heavy environments, review the implementation, run your own tests, and apply your organisation’s supply-chain policies.

02. Prerequisites

What you need before installing

OmniAssert is intended for test projects in the current .NET ecosystem. Basic assertions need only the package and a compatible target framework. Compile-time boolean diagnostics require the newer compiler features used by C# interceptors.

For normal package usage

  • A test project targeting net10.0 or a compatible target framework.
  • The .NET SDK installed locally and available on your PATH.
  • A test framework such as xUnit, NUnit, or MSTest. OmniAssert is framework-agnostic.
  • NuGet access to install the OmniAssert package.

For interceptor diagnostics

  • .NET 10 SDK.
  • C# 14 or later.
  • InterceptorsNamespaces configured if your project pins language settings manually.
  • No opt-out property such as OmniAssertDisableVerifyInterceptors set to true.

03. Installation

Install and configure OmniAssert

Install the package in your test project, import the OmniAssert namespace, then write assertions using .Verify(), .VerifyNullable(), .VerifyExpression(), and the exception/file helpers.

Add the NuGet package

Use the .NET CLI from the directory that contains your test project file.

dotnet add package OmniAssert --version 1.2.0

Or add the package reference manually

If you prefer editing the project file directly, add a package reference to your test project.

<ItemGroup>
  <PackageReference Include="OmniAssert" Version="1.2.0" />
</ItemGroup>

Import the namespace

Bring the extension methods into scope in each test file, or configure a global using for your test project.

using OmniAssert;

Check interceptor settings if needed

Interceptors are bundled and enabled by default. If your project pins language settings, ensure C# 14 and the generated namespace are configured. 

<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>
  <LangVersion>14</LangVersion>
  <InterceptorsNamespaces>$(InterceptorsNamespaces);OmniAssert.Generated</InterceptorsNamespaces>
</PropertyGroup>

To opt out of compile-time boolean diagnostics, set:

<PropertyGroup>
  <OmniAssertDisableVerifyInterceptors>true</OmniAssertDisableVerifyInterceptors>
</PropertyGroup>

04. Quick start

Your first OmniAssert test

This small xUnit example validates a user object. The same assertion style works with other .NET test frameworks because OmniAssert throws assertion exceptions directly. 

using OmniAssert;

public sealed record User(int Id, string Email, bool IsActive);

public class UserTests
{
    [Fact]
    public void Should_validate_user()
    {
        var user = new User(42, "ada@example.com", true);

        user.Id.Verify().ToBeGreaterThan(0);
        user.Email.Verify().ToContain("@");
        user.Email.Verify().ToEndWith(".com");
        user.IsActive.Verify().ToBeTrue();
    }
}
How to read the API: start with the value under test, call Verify(), then choose the assertion that describes the expected behaviour. For boolean conditions, use either flag.Verify().ToBeTrue() or (count > 0).VerifyExpression().

05. Advanced usage

Use OmniAssert across real test suites

OmniAssert covers common assertion categories and a few deeper workflows: soft assertions, object graph comparison, exception verification, unordered collection equivalence, span assertions, and compile-time boolean diagnostics.

Numbers ToBe, NotToBe, ToBeGreaterThan, ToBeInRange, ToBeApproximately, ToBeOneOf
Strings ToContain, ToStartWith, ToEndWith, ToMatch, ToBeIgnoringCase, length and whitespace assertions
Collections and spans ToHaveCount, ToBeUnique, AllSatisfy, AnySatisfy, NoneSatisfy, ordering checks, unordered equivalence
Objects ToBe for reference identity, ToBeOfType<T>, ToBeAssignableTo<T>, ToBeEquivalentTo for structural comparison
Exceptions and tasks Throws<T>, ThrowsAsync<T>, NotThrow, NotThrowAsync, message checks, inner exception checks, timeout checks
Other types Date/time, TimeSpan, Guid, Enum, Uri, nullable values, files, and directories

Soft assertions with AssertionScope

Use a scope when one test should report every related failure together. Instead of stopping at the first failed assertion, OmniAssert collects failures and throws when the scope is disposed. 

using OmniAssert;

[Fact]
public void Should_validate_imported_customer()
{
    var customer = ImportCustomer();

    using (new AssertionScope())
    {
        customer.Id.Verify().ToBeGreaterThan(0);
        customer.Name.Verify().NotToBeNullOrWhiteSpace();
        customer.Email.Verify().ToMatch(@"^[^@\s]+@[^@\s]+\.[^@\s]+$");
        customer.Roles.Verify().ToContain("Reader");
    }
}

Deep object comparison

ToBeEquivalentTo compares object graphs recursively and reports the path where expected and actual values differ. When you call ToBeEquivalentTo on a collection directly, OmniAssert treats the expected and actual sequences as multisets, so the same elements can appear in a different order. 

var expected = new InvoiceDto
{
    Number = "INV-2026-001",
    Currency = "GBP",
    Lines =
    [
        new InvoiceLineDto("Consulting", 2, 750m),
        new InvoiceLineDto("Support", 1, 125m)
    ]
};

var actual = await client.GetInvoiceAsync("INV-2026-001");

actual.Verify().ToBeEquivalentTo(expected);
actual.Lines.Verify().ToBeEquivalentTo(expected.Lines);

Collections, predicates, and ordering

var results = await search.IndexAsync(documents);

results.Verify().ToHaveCount(documents.Count);
results.Verify().ToBeUnique();
results.Verify().AllSatisfy(result => result.Score > 0);
results.Verify().AnySatisfy(result => result.Title.Contains("OmniAssert"));
results.Verify().NoneSatisfy(result => result.Status == "Failed");
results.Verify().ToBeInDescendingOrder(result => result.Score);

Exception and asynchronous assertions

Action syncAct = () => parser.Parse("");
syncAct.Throws<ArgumentException>()
    .WithMessageContaining("input");

Func<Task> asyncAct = () => service.CreateUserAsync(invalidRequest);
await asyncAct.ThrowsAsync<ValidationException>();

await TimeSpan.FromSeconds(2).CompleteWithin(
    () => service.RebuildProjectionAsync());

Compile-time boolean diagnostics

VerifyExpression() is useful when the shape of the boolean condition matters. With interceptors active, the generator rewrites supported call sites so failure output can include operand snapshots. 

var count = orders.Count;
var limit = 10;
var hasPriorityOrder = orders.Any(order => order.Priority == Priority.High);

(count > 0 && count <= limit && hasPriorityOrder).VerifyExpression();

Advanced operand-capture rewrite from source files

Repository-based or experimental setups can provide a source file as an AdditionalFiles item and exclude it from direct compilation. The generator emits the rewritten compilation unit with richer operand capture. This is not required for normal NuGet consumers. 

<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
    <LangVersion>14</LangVersion>
    <InterceptorsNamespaces>$(InterceptorsNamespaces);OmniAssert.Generated</InterceptorsNamespaces>
  </PropertyGroup>

  <ItemGroup>
    <AdditionalFiles Include="Program.cs" />
    <Compile Remove="Program.cs" />
  </ItemGroup>

  <ItemGroup>
    <ProjectReference Include="..\..\OmniAssert.Core\OmniAssert.Core.csproj" />
    <ProjectReference Include="..\..\OmniAssert.Generator\OmniAssert.Generator.csproj"
                      OutputItemType="Analyzer"
                      ReferenceOutputAssembly="false" />
  </ItemGroup>
</Project>

File system and URI checks

"appsettings.test.json".FileExists()
    .HaveContent(expectedJson);

"artifacts".DirectoryExists()
    .BeEmpty();

06. Troubleshooting

Frequently asked questions

Most setup issues come from target framework mismatches, missing namespace imports, or projects that pin language/compiler options.

Why does .Verify() not appear in IntelliSense?

Confirm that the package is referenced by the test project and that using OmniAssert; is in scope. If you use central package management or shared props files, restore packages and reload the project.

Why are interceptor diagnostics not showing operand values?

Check that the project builds with the .NET 10 SDK and C# 14, and that OmniAssertDisableVerifyInterceptors is not set to true. If you pin language settings, add OmniAssert.Generated to InterceptorsNamespaces.

Should I use Verify().ToBeTrue() or VerifyExpression()?

Use Verify().ToBeTrue() for simple boolean values and consistency with other fluent assertions. Use VerifyExpression() when a compound condition benefits from expression capture and richer diagnostics.

Why did my collection equivalence assertion pass even though the order changed?

ToBeEquivalentTo for collections treats the expected and actual sequences as multisets. Use ordering assertions such as ToBeInAscendingOrder() or compare an exact sequence when order is a behaviour requirement.

Why do I see several failures in one exception?

The assertions were probably run inside an AssertionScope. This is expected: the scope collects failures and throws an aggregate result when disposed.

Can I use OmniAssert with NUnit, xUnit, or MSTest?

Yes. OmniAssert is a standalone assertion library and is not tied to a specific test runner. Import the namespace and use it from whichever .NET test framework your project already uses.
